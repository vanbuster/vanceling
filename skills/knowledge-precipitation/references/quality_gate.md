# 质量门禁

> 最后更新：2026-05-25
> 用途：为新文档执行强制检查，确保知识库质量

## 质量门禁分层

### L1：硬性规则检查（自动扫描层）

**执行时机**：文档保存前

**检查项**：

#### L1-1：命名规范检查

**检查内容**：
- 文件名是否包含日期（YYYY-MM-DD）？
- 日期格式是否正确？
- 核心主题是否为中文（技术术语可保留英文）？
- 扩展名是否正确（.md 或 .txt）？
- 核心主题长度是否在 5-30 字之间？
- 文件名是否重复？

**通过标准**：6 项全部通过

**失败处理**：
- 提示具体哪项不通过
- 自动建议修正方案
- 用户确认后修正

**示例**：
```
❌ 命名规范检查失败：
  - 核心主题无中文

  建议修正：卡兹克风格深度分析_2026-05-25.md

  确认修正？(Y/n)
```

---

#### L1-2：分类正确性检查

**检查内容**：
- 分类是否为 5 个分类之一（产品分析/技术方案/项目分析/参考资料/工具研究）？
- 分类是否与文档内容匹配？
- 二级标签是否正确？

**通过标准**：3 项全部通过

**失败处理**：
- 提示具体哪项不通过
- 建议正确的分类
- 用户确认后修正

**示例**：
```
❌ 分类正确性检查失败：
  - 分类与内容不匹配

  当前分类：技术方案
  建议分类：产品分析

  确认修正？(Y/n)
```

---

#### L1-3：结构完整性检查

**检查内容**：
- 文档是否有明确标题？
- 文档是否有章节结构（## 一级标题）？
- 文档是否完整（无"待补充"、"TODO"标记）？
- 文档是否过短（少于 500 字）？

**通过标准**：4 项全部通过

**失败处理**：
- 提示具体哪项不通过
- 建议完善内容
- 用户确认后完善

**示例**：
```
❌ 结构完整性检查失败：
  - 文档过短（320 字）

  建议补充：
    1. 添加结论或总结
    2. 添加案例或实例
    3. 添加背景或上下文

  继续保存？(Y/n)
```

---

#### L1-4：文件路径检查

**检查内容**：
- 文件路径是否为 `docs/<分类>/<文件名>`？
- 文件夹是否存在？
- 文件是否已成功保存？

**通过标准**：3 项全部通过

**失败处理**：
- 自动创建文件夹（如果不存在）
- 重新保存文件（如果保存失败）
- 提示用户（如果无法自动处理）

**示例**：
```
❌ 文件路径检查失败：
  - 文件夹不存在

  自动处理：创建文件夹 docs/产品分析/

  继续保存？(Y/n)
```

---

#### L1-5：索引更新检查

**检查内容**：
- INDEX.md 是否已更新？
- 文档是否已添加到对应分类表格？
- 时间线是否已更新？
- 统计信息是否已更新？

**通过标准**：4 项全部通过

**失败处理**：
- 自动更新 INDEX.md
- 提示更新内容
- 用户确认后完成

**示例**：
```
❌ 索引更新检查失败：
  - INDEX.md 未更新

  自动处理：更新 INDEX.md
  - 添加到产品分析表格
  - 更新时间线
  - 更新统计信息

  继续保存？(Y/n)
```

---

#### L1-6：迭代记录检查

**检查内容**：
- 自检与迭代.md 是否已记录？
- 操作类型是否正确（新增/修正/归档/删除/移动）？
- 变化说明是否完整？
- 影响范围是否注明？

**通过标准**：4 项全部通过

**失败处理**：
- 自动记录到 自检与迭代.md
- 提示记录内容
- 用户确认后完成

**示例**：
```
❌ 迭代记录检查失败：
  - 自检与迭代.md 未记录

  自动处理：记录到 自检与迭代.md
  - 操作类型：新增
  - 文档名称：卡兹克风格深度分析_2026-05-25.md
  - 分类：产品分析
  - 变化说明：分析卡兹克写作风格...

  继续保存？(Y/n)
```

---

### L1 通过标准

**规则**：6 个检查项全部通过

**输出格式**：
```
✅ L1 硬性规则检查通过
  ✅ 命名规范检查
```

**规则**：用户对所有确认项选择 "Y"

### L3：定期自检层（每月）

**执行时机**：每月第一个周一

**检查项**：

#### L3-1：有效性检查

**检查内容**：
- 内容是否仍然有效？
- 是否有被后续实践推翻的结论？
- 是否有技术或行业变化导致内容过时？

**通过标准**：全部有效

**失败处理**：
- 标记为"可能过时"
- 询问用户：更新/归档/删除

---

#### L3-2：准确性检查

**检查内容**：
- 事实是否有误？
- 数据是否准确？
- 引用是否正确？

**通过标准**：全部准确

**失败处理**：
- 修正错误内容
- 记录到 自检与迭代.md

---

#### L3-3：完整性检查

**检查内容**：
- 是否缺少关键信息？
- 是否有断链或引用缺失？
- 是否有未完成的章节？

**通过标准**：全部完整

**失败处理**：
- 补充缺失内容
- 记录到 自检与迭代.md

---

#### L3-4：复用性检查

**检查内容**：
- 是否有复用价值？
- 是否被引用过？
- 是否有后续文档依赖？

**通过标准**：有复用价值

**失败处理**：
- 归档（3 个月未更新但可能有复用价值）
- 删除（6 个月未访问且无复用价值）

---

#### L3-5：分类检查

**检查内容**：
- 是否在正确分类？
- 分类是否需要调整？
- 是否有二级分类的需求？

**通过标准**：分类正确

**失败处理**：
- 移动到正确分类
- 更新 INDEX.md
- 记录到 自检与迭代.md

---

### L3 通过标准

**规则**：
- 有效、准确、完整、有复用价值、分类正确
- 或：已处理（更新/归档/删除）

**输出格式**：
```
自检报告 - 2026-06-02
━━━━━━━━━━━━━━━━━━━━━━

检查文档：12 个
✅ 有效：12
✅ 准确：12
✅ 完整：12
✅ 有复用价值：12
✅ 分类正确：12

处理结果：
  - 无需处理

下次自检：2026-07-02
```

## 质量门禁执行流程

### 完整流程

```
开始
  ↓
L1 硬性规则检查（6 项）
  ↓
L1 通过？
  ├─ 否 → 自动修正 → 用户确认 → 重新检查
  └─ 是 → 继续
  ↓
L2 人工确认层（3 项）
  ↓
L2 通过？
  ├─ 否 → 用户调整 → 重新确认
  └─ 是 → 继续
  ↓
保存文档
  ↓
更新 INDEX.md
  ↓
记录到 自检与迭代.md
  ↓
输出总结
  ↓
结束
```

---

### L3 自检流程（每月）

```
开始
  ↓
扫描所有文档
  ↓
L3 检查（5 项）
  ↓
发现问题？
  ├─ 是 → 询问用户处理方式（更新/归档/删除）
  │     → 执行处理
  │     → 记录到 自检与迭代.md
  └─ 否 → 继续
  ↓
生成自检报告
  ↓
结束
```

## 质量门禁配置

### 可配置项

**L1 配置**：
- 核心主题最小长度：5 字
- 核心主题最大长度：30 字
- 文档最小字数：500 字
- 允许的扩展名：.md, .txt

**L2 配置**：
- 是否需要人工确认：是/否
- 确认项：分类/文件名/关联关系

**L3 配置**：
- 自检频率：每月第一个周一
- 归档阈值：3 个月未更新
- 删除阈值：6 个月未访问

## 最佳实践

### DO ✅

- L1 检查必须全部通过才能保存
- L2 确认可选，但建议执行
- L3 自检每月执行，不要跳过
- 所有检查结果都记录到 自检与迭代.md

### DON'T ❌

- 不要跳过 L1 检查
- 不要强行保存不通过的文档
- 不要忽略 L3 自检
- 不要删除检查记录

## 质量门禁的意义

**目的**：
- 确保知识库的质量
- 避免低质量内容进入知识库
- 提高知识库的可复用性
- 降低知识库的维护成本

**原则**：
- 宁可拒绝，也不降标准
- 宁可延期，也不草率
- 宁可返工，也不留隐患

---

**注意**：质量门禁不是为了限制用户，而是为了保护知识库。好的质量门禁应该让用户感受到价值，而不是感受到束缚。
