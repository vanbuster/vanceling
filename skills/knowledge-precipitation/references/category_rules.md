# 分类规则

> 最后更新：2026-05-25
> 用途：为新文档自动判断分类提供规则参考

## 一级分类决策树

```
开始
  ↓
文档内容关于什么？
  ├─ 产品/写作/风格/方法论 → 产品分析
  ├─ 技术/架构/配置/方案 → 技术方案
  ├─ 项目/竞品/实践/拆解 → 项目分析
  ├─ 原始素材/PDF摘录/第三方资料 → 参考资料
  └─ 工具/格式/最佳实践 → 工具研究
```

## 详细分类规则

### 产品分析

**判断标准**：
- 内容是否涉及产品设计方法论？
- 是否分析某个写作风格或表达方式？
- 是否提供创作流程或生产流程？
- 是否讨论产品思维或产品决策？

**关键词**：
- 产品、写作、风格、方法、流程、工作流
- 卡兹克、小红书、公众号
- 风格分析、方法论、创作流程

**二级标签**：
- `#方法论` - 产品方法论、分析方法论、写作方法论
- `#风格分析` - 他人风格提炼、写作风格分析
- `#工作流` - 创作流程、生产流程、内容生产

**示例**：
- 卡兹克风格深度分析_2026-05-25.md（`#风格分析` `#方法论`）
- 小红书创作工作流_2026-05-25.md（`#工作流` `#方法论`）

---

**判断标准**：
- 内容是否涉及技术选型？
- 是否讨论系统架构或技术架构？
- 是否提供配置方案或实施方案？
- 是否解决某个技术问题？

**关键词**：
- 技术、架构、配置、方案、选型
- Hermes、TUI、Prompt
- 技术方案、架构设计、配置说明

**二级标签**：
- `#配置` - 系统配置、环境配置、工具配置
- `#架构` - 系统架构、技术架构、架构设计

**示例**：
- hermes-sessions-config-complete.md（`#配置`）
- hermes-sessions-management-plan.md（`#方案`）
- hermes-sessions-quick-start.md（`#配置`）

**关键词**：

## 跨分类处理

### 情况 1：内容跨多个分类

**处理方式**：
- 选择最核心的分类
- 其他分类用关联关系标注

**示例**：
- 一个文档既分析卡兹克风格（产品分析），又提供了小红书创作工作流（产品分析）
- 分类：产品分析
- 关联：无（都是产品分析）

---

### 情况 2：技术方案 + 项目分析

**处理方式**：
- 如果重点是技术实现 → 技术方案
- 如果重点是项目拆解 → 项目分析
- 关联标注：技术方案 → 项目分析（或反之）

**示例**：
- "Coze工作流搭建"可以是技术方案（实现方案）或项目分析（项目总结）
- 如果重点是"如何搭建" → 技术方案
- 如果重点是"搭建后的总结" → 项目分析

---

### 情况 3：不确定分类

**处理方式**：
- 询问用户："这个文档应该属于哪个分类？"
- 提供建议：根据内容，可能是A或B分类
- 用户确认后执行

---

## 关联关系判断

### 依赖关系（→）

**定义**：文档A依赖文档B，理解A需要先看B

**判断标准**：
- 文档A引用文档B的内容
- 文档A在文档B的基础上扩展
- 文档A使用文档B的方法论

**示例**：
- hermes-sessions-management-plan.md（方案）→ hermes-sessions-config-complete.md（配置）
  - 方案依赖配置
- xiaohongshu_formatting_research.md（格式研究）→ 小红书创作工作流_2026-05-25.md（工作流）
  - 工作流依赖格式研究

---

### 演进关系（→）

**定义**：文档A是文档B的演进版本

**判断标准**：
- 文档A是文档B的更新版
- 文档A在文档B基础上做了重大改进
- 文档A取代文档B

**示例**：
- 如果有"小红书创作工作流_v2_2026-06-01.md"，它会关联到"小红书创作工作流_2026-05-25.md"

---

### 补充关系（→）

**定义**：文档A补充文档B，提供更多细节

**判断标准**：
- 文档A提供文档B的补充信息
- 文档A是文档B的案例或实例
- 文档A是文档B的某个方面的展开

**示例**：
- pdf1_xingye_raw.txt（原始素材）→ PDF产品拆解_结构化摘要.md（摘录）
  - 摘录补充原始素材
- 机器人综合实践_项目架构分析_2026-05-20.md（拆解）→ coze工作流搭建完成报告.md（实践报告）
  - 实践报告补充拆解

## 自动分类算法

### 算法 1：关键词匹配

```
输入：文档内容
输出：建议分类

步骤：
1. 提取文档中的关键词
2. 匹配每个分类的关键词表
3. 计算匹配得分（匹配关键词数 / 该分类关键词总数）
4. 选择得分最高的分类
5. 如果多个分类得分相近，询问用户
```

---

### 算法 2：标题分析

```
输入：文档标题
输出：建议分类

规则：
- 包含"风格分析" → 产品分析
- 包含"工作流" → 产品分析
- 包含"方案" → 技术方案
- 包含"配置" → 技术方案
- 包含"拆解" → 项目分析
- 包含"实践报告" → 项目分析
- 包含"raw" → 参考资料
- 包含"研究" → 工具研究
- 包含"调研" → 工具研究
```

## 最佳实践

### DO ✅

- 仔细阅读文档内容后再判断分类
- 不确定时询问用户，不要强行分类
- 关联关系要准确，不要建立错误的关联
- 二级标签要精确，不要泛泛而谈

### DON'T ❌

- 不要只看标题就分类（要看内容）
- 不要把跨分类内容强行塞进一个分类
- 不要遗漏重要的关联关系
- 不要用过于宽泛的二级标签（如用#方法论代替#风格分析）

---

**注意**：分类的目的是让知识库有结构，不是为了分类而分类。正确的分类能让用户快速找到内容，错误的分类会让知识库混乱。
