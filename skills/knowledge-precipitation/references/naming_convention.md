# 命名规范

> 最后更新：2026-05-25
> 用途：为新文档自动生成文件名提供规范参考

## 标准格式

```
核心主题_YYYY-MM-DD.ext
```

---

## 格式详解

### 核心

**规则**：
- 用中文描述文档的核心主题
- 技术术语保留英文（如 Hermes、Coze、小红书）
- 长度：5-30 个字（含标点）

**示例**：
- ✅ 卡兹克风格深度分析
- ✅ 小红书创作工作流
- ✅ hermes-sessions-config-complete（已存在，沿用原名）
- ✅ 机器人综合实践_项目架构分析
- ❌ Analysis（无中文）
- ❌ 卡兹克风格分析（可以，但不如"深度分析"精确）
- ❌ 关于卡兹克风格的深度分析（太长，啰嗦）

**提取方法**：
1. 读取文档标题（如果有的话）
2. 如果没有标题，读取第一段内容，提取核心主题
3. 询问用户："核心主题是什么？"
4. 综合判断后确定核心主题

---

### 日期

**格式**：YYYY-MM-DD

**规则**：
- 年份：4 位数（2026）
- 月份：2 位数（05）
- 日期：2 位数（25）
- 使用创建日期，不是修改日期

**示例**：
- ✅ 2026-05-25
- ✅ 2026-12-01
- ❌ 2026-5-25（月份不是 2 位数）
- ❌ 05-25（缺少年份）
- ❌ 20260525（缺少分隔符）

**提取方法**：
1. 检查文档内容是否包含日期（如"2026-05-25"、"2026年5月25日"）
2. 如果没有，使用当前日期
3. 格式化为 YYYY-MM-DD

---

### 扩展名

**规则**：
- `.md`：Markdown 文档（结构化的分析、方案、报告）
- `.txt`：纯文本（原始素材、对话记录、未经整理的内容）

**示例**：
- ✅ 卡兹克风格深度分析_2026-05-25.md
- ✅ pdf1_xingye_raw.txt
- ❌ 卡兹克风格深度分析_2026-05-25.txt（应该是 .md）
- ❌ 卡兹克风格深度分析_2026-05-25.doc（不支持）

**提取方法**：
1. 检查现有文件扩展名
2. 如果是结构化的 Markdown 内容 → `.md`
3. 如果是原始素材、对话记录 → `.txt`
4. 询问用户（如果不确定）

---

## 自动命名算法

### 算法 1：基于标题

```
输入：文档标题、当前日期
输出：建议文件名

步骤：
1. 检查标题是否为中文
2. 如果是，直接使用标题
3. 如果不是，询问用户："核心主题是什么？"
4. 添加日期（YYYY-MM-DD）
5. 添加扩展名（.md 或 .txt）
```

**示例**：
- 标题："卡兹克风格深度分析" → `卡兹克风格深度分析_2026-05-25.md`
- 标题："AI Music Blog v2 Architecture" → 询问用户 → `AI音乐博客v2架构分析_2026-05-25.md`

---

### 算法 2：基于内容

```
输入：文档内容、当前日期
输出：建议文件名

步骤：
1. 提取第一段内容
2. 使用关键词提取算法（TF-IDF 或 TextRank）
3. 选择前 3 个关键词
4. 组合成核心主题（最多 30 字）
5. 添加日期（YYYY-MM-DD）
6. 添加扩展名（.md 或 .txt）
7. 询问用户确认
```

**示例**：
- 第一段："本文分析了卡兹克的写作风格，提炼出核心要素，提出去模板化方案。"
- 关键词：卡兹克、写作风格、去模板化
- 核心主题：卡兹克写作风格去模板化方案
- 建议文件名：`卡兹克写作风格去模板化方案_2026-05-25.md`

---

### 算法 3：基于用户输入

```
输入：用户输入的核心主题、当前日期
输出：建议文件名

步骤：
1. 检查用户输入是否符合规范（中文、5-30 字）
2. 如果不符合，提示用户修改
3. 添加日期（YYYY-MM-DD）
4. 添加扩展名（.md 或 .txt）
5. 询问用户确认
```

**示例**：
- 用户输入："卡兹克风格"
- 检查：符合规范
- 建议文件名：`卡兹克风格_2026-05-25.md`

## 命名检查规则

### 规则 1：核心主题必须是中文

**检查逻辑**：
```
如果核心主题不包含中文：
  提示：核心主题必须使用中文（技术术语可保留英文）
  询问：请重新输入核心主题
```

**示例**：
- ❌ `Analysis_2026-05-25.md`（无中文）
- ✅ `深度分析_2026-05-25.md`（有中文）

---

### 规则 2：日期格式必须正确

**检查逻辑**：
```
如果日期不符合 YYYY-MM-DD 格式：
  提示：日期格式错误，应为 YYYY-MM-DD
  自动修正为当前日期
```

**示例**：
- ❌ `深度分析_2026-5-25.md`（月份不是 2 位数）
- ✅ `深度分析_2026-05-25.md`（正确）

---

### 规则 3：扩展名必须正确

**检查逻辑**：
```
如果扩展名不是 .md 或 .txt：
  提示：扩展名必须是 .md 或 .txt
  根据内容类型自动修正
```

**示例**：
- ❌ `深度分析_2026-05-25.doc`（不支持）
- ✅ `深度分析_2026-05-25.md`（正确）

---

### 规则 4：文件名不能过长

**检查逻辑**：
```
如果核心主题超过 30 字：
  提示：核心主题过长，请精简到 30 字以内
  提供建议：截取前 30 字，或询问用户精简
```

**示例**：
- ❌ `关于卡兹克写作风格的深度分析和去模板化方案建议_2026-05-25.md`（过长）
- ✅ `卡兹克写作风格去模板化方案_2026-05-25.md`（精简后）

---

### 规则 5：文件名不能重复

**检查逻辑**：
```
如果文件名已存在：
  提示：文件名已存在，请修改
  建议方案：
    1. 修改核心主题
    2. 添加版本号（v2）
    3. 更改日期
```

**示例**：
- 已存在：`卡兹克风格深度分析_2026-05-25.md`
- 新文档：`卡兹克风格深度分析_2026-05-25.md`（重复）
- 建议：
  1. `卡兹克风格分析v2_2026-05-25.md`
  2. `卡兹克风格分析_2026-05-26.md`

## 特殊情况处理

### 情况 1：沿用原有文件名

**适用场景**：
- 文档已存在，只是更新内容
- 文件名不符合规范，但用户不想改

**处理方式**：
- 保持原有文件名不变
- 在 INDEX.md 中标注"沿用原名"
- 在 自检与迭代.md 中记录

**示例**：
- 文档：`hermes-sessions-config-complete.md`
- 标注：沿用原名（不符合规范）

---

### 情况 2：版本迭代

**适用场景**：
- 文档更新，需要保留旧版本
- 文档重构，需要标记新版本

**处理方式**：
- 旧版本：`文档名_v1_YYYY-MM-DD.md`
- 新版本：`文档名_v2_YYYY-MM-DD.md`
- 在 INDEX.md 中建立演进关系

**示例**：
- v1：`小红书创作工作流_v1_2026-05-25.md`
- v2：`小红书创作工作流_v2_2026-06-01.md`
- 关联：v1 → v2

---

### 情况 3：临时文档

**适用场景**：
- 工作过程中的临时文件
- 需要明确过期时间的文档

**处理方式**：
- 文件名添加 `[临时]` 标记
- 在 自检与迭代.md 中记录过期时间
- 过期后自动删除或归档

**示例**：
- 文件名：`[临时]小红书文案草稿_2026-05-25.md`
- 过期时间：2026-06-01
- 处理：过期后删除

## 最佳实践

### DO ✅

- 核心主题要准确反映文档内容
- 日期使用创建日期，不是修改日期
- 扩展名根据内容类型选择（.md 或 .txt）
- 文件名重复时，优先考虑版本号

### DON'T ❌

- 不要使用英文作为核心主题（除非是技术术语）
- 不要让核心主题过长（超过 30 字）
- 不要混用不同日期格式
- 不要在核心主题中添加无关信息（如"草稿"、"最终版"）

---

## 与其他系统的集成

### 与 Memory 集成

如果用户有特殊的命名习惯：
- 记录到 Memory（如"用户喜欢用'深度分析'而非'分析'"）
- 下次生成文件名时自动应用

---

### 与 Obsidian 集成

如果文档需要同步到 Obsidian：
- 保持相同的文件名
- 使用 `[[]]` 双链格式
- 添加 Obsidian 属性（tags、created、updated）

---

**注意**：命名规范的目的不是限制用户，而是让知识库有统一的标准，便于管理和检索。规范应该服务于人，而不是人服务于规范。
